# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Murmur is a single-binary macOS menu-bar app (Swift + SwiftUI + WKWebView) that plays YouTube audio in the background, with an optional chromeless floating video window. macOS 13+, Swift 5.9+, no external dependencies — `Package.swift` declares only the `Murmur` executable target rooted at `Sources/Murmur`.

## Commands

- **Run locally:** `swift run -c release` (debug builds work but `-c release` is what the README documents and what build-app.sh ships).
- **Build a shareable `.app` + zip:** `./build-app.sh` (writes to `dist/`). Flags: `--sign` for ad-hoc codesign, `--open` to reveal in Finder.
- **Regenerate the app icon:** `swift make-icon.swift` rewrites `icon.png` from the in-app palette; rerun `./build-app.sh` to rebake `AppIcon.icns` into the bundle.
- **Regenerate social card:** `swift make-social-card.swift` overwrites `social-preview.png`.

There is **no test target** and no linter configured — `Package.swift` only defines the executable. Don't invent `swift test` instructions.

## Singleton vs `@EnvironmentObject` rule

| Pattern | When to use |
|---|---|
| `.shared` **singleton** | Pure data stores with no constructor dependencies (UserDefaults-backed, network-fetching). Examples: `FavoritesStore`, `PlayedVideoHistoryStore`, `UserPlaylistsStore`, `PlaylistStore`, `TrendingRegionStore`, `LastSessionStore`, `SearchHistoryStore`, `APIKeyStore`, `UpdateChecker`, `ChannelFavoritesStore`, `QuotaTracker`, `PlaybackQueue`, `WindowOpenerBridge`, `YouTubeSearchState`. Views reach for them as `@ObservedObject private var foo = FooStore.shared`. |
| **`@EnvironmentObject`** | Things with constructor dependencies or AppKit resources that AppDelegate must own. Examples: `PlayerController` (owns `WKWebView`), `VideoWindowController` (owns `NSWindow`), `MixerEngine` (owns the audio engine and must `try start()`), `BoothLauncher` / `RecordingsLauncher` (wrap controllers with constructor args). |

**Don't mix both for the same type.** A store should be either reachable via `.shared` OR injected via env — never both. The two-paths-to-the-same-state pattern was confusing in earlier iterations.

## Architecture (the pieces that span files)

The non-obvious design decisions live across `Murmur.swift`, `AppDelegate.swift`, `VideoWindowController.swift`, and `PlayerController.swift`. Read these together before changing playback or window behavior:

- **The WKWebView must always live inside a real on-screen-eligible NSWindow.** WebKit suspends media playback on 0×0 / detached views, so `VideoWindowController` keeps the window alive at all times — when the user wants "audio only" it's parked at `(-3000, -3000)` with a `.borderless` styleMask. Toggling video on swaps the styleMask to `[.titled, .closable, .resizable, .fullSizeContentView]` and lifts to `.floating`. Don't "destroy and recreate" the webview on toggle — you'll kill audio.
- **Pin-to-all-Spaces is a separate axis from window visibility.** `VideoWindowController.isPinned` (persisted under `youtube-audio-widget.videoWindow.pinned`) toggles `collectionBehavior` between `[.fullScreenAuxiliary]` (default — visible only on its origin Space) and `[.canJoinAllSpaces, .stationary, .fullScreenAuxiliary]` (follows the user when they switch Mission Control desktops, including into full-screen apps). This is **Spaces**, not **physical displays** — moving across monitors still needs a manual drag. The behavior is applied in `init` from UserDefaults before the user ever toggles video on, so the first show already respects the saved preference.
- **Window dragging is layered.** `WindowDragOverlay` is a transparent `NSView` on top of the WKWebView that calls `window.performDrag(with:)` in `mouseDown`. WKWebView swallows mouseDown before AppKit checks `isMovableByWindowBackground`, so that flag alone is unreliable — both mechanisms are kept on purpose.
- **YouTube embed is iframe + postMessage, not the JS `YT.Player` constructor.** `PlayerController.loadPlayer` builds an HTML page that hosts a `youtube-nocookie.com/embed/<id>` iframe and talks to it via the IFrame API's `postMessage` / `listening` handshake. Swift→JS goes through `evaluateJavaScript("window.ytCmd(...)")`; JS→Swift goes through `webkit.messageHandlers.cb.postMessage` (handled by `ScriptHandler`). Using `YT.Player` inside `loadHTMLString` runs into origin/baseURL issues — don't switch to it.
- **Two layers of "loading mask" hide YouTube branding flashes.** A Swift-side opaque NSView (`VideoWindowController.loadingMask`, fired from `controller.onWillLoadStream` → `flashLoadingMask()`) covers the webview during reload, and an in-page `#cover` div covers the iframe whenever the player isn't actively Playing (states ≠ 1, except Buffering which is treated as in-flight playback). The first hide waits ~1.5s past YouTube's startup branding overlay; subsequent hides on resume are immediate. The cover also masks YouTube's paused-state overlay (Topic-channel "now playing" card + center prev/play/next pills), so don't remove the cover from the DOM after first fade. Keep both layers — each catches a different flash (Swift-side: WKWebView reload; JS-side: YouTube's startup + paused-state branding).
- **Iframe pointer events are disabled** (`iframe { pointer-events: none }`) so YouTube's title/share/Watch-on-YouTube hover overlays never appear. All controls come from Swift; the iframe is never meant to be clicked.
- **`navigationDelegate.didFinish` has a 1.5s "force-ready" fallback.** Some live streams never fire `onReady` through the IFrame API channel handshake. After 1.5s `isReady` is forced true so the user can hit play manually. If you change autoplay logic, preserve this fallback.
- **`applicationShouldTerminateAfterLastWindowClosed → false`** and the close button is intercepted in `windowShouldClose` (hides instead of closing). The menu-bar panel and floating video window are NSWindows; closing them must not quit the app — only the panel's "Quit" button calls `NSApp.terminate`.
- **Menu-bar icon is AppKit `NSStatusItem` (in `AppDelegate`), not SwiftUI `MenuBarExtra`.** We tried `MenuBarExtra(.window)` (chromeless panel on click) and `.menu` (dropdown) but neither matches the desired UX of "click the icon → main window opens, period." SwiftUI offers no "primary action" style for `MenuBarExtra`. Switching to `NSStatusItem` with a `button.action = #selector(statusItemClicked(_:))` gives us a true single-click handler. The action calls `NSApp.activate(ignoringOtherApps: true)` then `WindowOpenerBridge.shared.openMain?()` to bring the main `Window` scene up. The bridge is a tiny singleton holding a captured `openWindow` closure — populated on `MainWindowView`'s `.onAppear` because `@Environment(\.openWindow)` only exists inside SwiftUI view bodies, and AppDelegate has no environment of its own. Captured closures remain valid for the life of the `App` (they reference SwiftUI's window-management machinery, not the disposed view instance), so even after the user closes the main window we can reopen it from the status item. **Don't reintroduce `MenuBarExtra`** — the icon click would silently start showing a panel/menu again.
- **`AppDelegate` is attached via `@NSApplicationDelegateAdaptor` (`Murmur.swift`).** Owns `PlayerController`, `MixerEngine`, and the lazy `VideoWindowController` / `BoothWindowController` / `RecordingsWindowController` (+ their `BoothLauncher` / `RecordingsLauncher` wrappers). `applicationDidFinishLaunching` does the player setup, mixer start, Combine-sink installation (`historyCancellable`, `positionCancellable`, `userPlaylistCancellable`, `lastSessionCancellable`), `controller.onEnded` auto-advance handler, `restoreLastSession()`, the activation-policy launch contract (see the READ-BEFORE-TOUCHING block at the top of `AppDelegate.swift`), and `NSStatusItem` creation. The deep-link handler `application(_:open:)` for `murmur://` also lives here. **The video / booth / recordings windows are `lazy var` and force-touched inside `applicationDidFinishLaunching` (`_ = videoWindow; _ = booth; _ = recordings`)** so they're guaranteed initialized before SwiftUI evaluates the scene body (which reads `delegate.videoWindow` etc. as `EnvironmentObject`s) — without that, the body could observe nil-equivalent state on the very first interaction.
- **`var body: some SwiftUI.Scene` is fully qualified.** The project ships its own `struct Scene` (`Sources/Murmur/Scenes/Scene.swift`, a DJ booth preset model) which shadows SwiftUI's `Scene` protocol at the call site. Don't drop the `SwiftUI.` qualifier — the `MurmurApp.body` declaration won't compile without it.
- **`Murmur.swift`, not `main.swift`.** A file literally named `main.swift` is parsed as top-level code, which is incompatible with `@main` on an `App` struct (`'main' attribute cannot be used in a module that contains top-level code`). The entry point file is therefore `Sources/Murmur/Murmur.swift`. Don't rename it back.
- **`.murmurDismissPopoverSheets` Notification** still exists as a safety lever. With `MenuBarExtra`, the system tears down detached sheets on dismiss, so nothing posts the notification today. Posting it from anywhere flips every sheet `@State` binding to false.
- **Main UI is a `WindowGroup`; app ends up menu-bar-only (no Dock icon) via a brief `.regular → .accessory` dance at launch.** `MurmurApp.body` declares `WindowGroup("Murmur", id: "main")` as the first scene. **It must be `WindowGroup`, not `Window`** — per Apple's docs only `WindowGroup` is auto-presented on launch. We still get single-window behavior because `openWindow(id: "main")` brings the existing window to the front rather than creating duplicates. The activation-policy choreography is finicky: SwiftUI's auto-presentation requires `.regular` at launch (accessory apps don't get a default-opened window), but the user wants no Dock icon. So `AppDelegate.applicationDidFinishLaunching` sets `.regular`, SwiftUI auto-opens the WindowGroup, and `MainWindowView.onAppear` immediately demotes to `.accessory` — the Dock icon flash is microseconds (before the WindowServer composites a visible frame). **Info.plist must NOT carry `LSUIElement = true`** — that flag forces `.accessory` at launch, our runtime `.regular` call gets ignored, SwiftUI skips the auto-open, and the user sees nothing. The status item (`NSStatusItem` in `AppDelegate`) is the persistent re-entry point after the user closes the window; `MainWindowView.onAppear` also captures `openWindow(id: "main")` into `WindowOpenerBridge.shared.openMain` so the status item click handler can reopen the window. **`.onDisappear` does NOT flip back to `.regular`** — once we're menu-bar-only we stay that way for the rest of the session, including across window reopens.
- **Auxiliary sheets are `Window` scenes, not `.sheet(isPresented:)` modifiers.** Each former sheet (queue, playlist, my playlists, YouTube search, API key, lyrics, World Cup) is declared as a top-level `Window` scene in `MurmurApp.body` and opened from `ContentView` via `@Environment(\.openWindow)`'s `openWindow(id:)` (window IDs: `queue`, `playlist`, `user-playlists`, `search`, `api-key`, `lyrics`, `world-cup`; the `api-key` scene is titled "Settings" and hosts launch-at-login + SponsorBlock + ListenBrainz + a button to the `stats` window). The `lyrics` window is conditionally surfaced — the header button only appears when `controller.categoryHint == .music`. This fixes two things at once: (1) the close-button-closes-the-app bug — `dismiss()` inside a sheet hosted by `MenuBarExtra` would propagate up and dismiss the menu-bar panel itself; with Window scenes, `dismiss()` resolves to the sheet's own window. (2) The CleanMyMac side-by-side layout — windows live independently of the panel, can be dragged, moved between screens, and remain open after the panel closes. Each Window scene applies `.windowResizability(.contentSize)` so the system adopts the SwiftUI view's intrinsic frame (each sheet has its own `.frame(...)`) instead of macOS's default 800×600. Sheet input parameters that used to be constructor arguments (`onPick` closures, `initialMode` / `initialQuery`) are gone — sheets now read `@EnvironmentObject var controller: PlayerController` and call `controller.load(input:)` themselves before `dismiss()`. The only parameterized one (search needs a seed mode + query) reads from `YouTubeSearchState.shared` — the caller sets `mode`/`query` immediately before `openWindow(id: "search")`, and the search view's `.onAppear` consumes + clears the seed so subsequent opens land on default state. `WindowGroup(id:for:)` would let us pass values directly but is macOS 14+.

## State and persistence

Per-store UserDefaults keys, the last-session restore contract, and the user-vs-YouTube playlist split now live in `Sources/Murmur/Ambient/CLAUDE.md`. Read that before changing any persistence behavior.

Cross-cutting bits that stay here:

- **Discover catalog** is hard-coded in `ContentView.swift` (`Self.catalog`). Adding categories/items there is the documented extension point — see `.claude/commands/add-discover-item.md` for the slash-command path. Stream IDs go stale when channels restart — that's expected.
- **Default video ID** is `kDefaultVideoID` at the top of `PlayerController.swift`. `ContentView`'s "Featured → Claude FM" entry references the same constant; keep them tied.
- The Combine sinks that drive persistence (`historyCancellable`, `positionCancellable`, `userPlaylistCancellable`, `lastSessionCancellable`) are installed in `AppDelegate.applicationDidFinishLaunching` **before** `restoreLastSession()` runs. Order is load-bearing — see the Ambient CLAUDE.md.

## App lifecycle / packaging

- Runs as `LSUIElement` (no Dock icon) — set both via `Info.plist` in `build-app.sh` and `NSApp.setActivationPolicy(.accessory)` inside `AppDelegate.applicationDidFinishLaunching` (the bundled app reads `Info.plist`; `swift run` doesn't, so the explicit call covers the dev-binary case). Both are needed.
- **`murmur://` URL scheme** is registered in `Info.plist`'s `CFBundleURLTypes` (added by `build-app.sh`) and handled by `AppDelegate.application(_:open:)`. Format: `murmur://play?v=<videoID>[&list=<playlistID>]`. Used by the popover's share menu so recipients with Murmur installed open straight into playback. URL handling only works in the built `.app` — `swift run` doesn't load Info.plist, so deep links won't reach the dev binary unless you register it with Launch Services manually.
- The popover is `.transient` and closes on outside click; the floating video window is independent and stays visible.
- `build-app.sh` produces an unsigned (or ad-hoc-signed) bundle. Recipients hit Gatekeeper on first launch — README documents the right-click → Open workaround and the `xattr -dr com.apple.quarantine` fallback.
- **Update notifications** come from `UpdateChecker` (`Ambient/UpdateChecker.swift`), which polls `https://api.github.com/repos/ketok-id/murmur-music/releases/latest` on launch + every 6 hours. When the latest GitHub release tag is numerically greater than `CFBundleShortVersionString`, the popover footer's version label flips from a dim `v…` to an accent-colored tappable badge. Clicking it **self-updates in place** — `UpdateChecker.downloadAndInstall()` → `SelfUpdater` downloads the release's `Murmur.zip`, de-quarantines it, swaps the bundle via a detached helper, and relaunches (see `Ambient/CLAUDE.md` → SelfUpdater). The badge shows a spinner while working and falls back to opening the release page if the install fails or the release shipped no zip asset. **The release flow must keep publishing `Murmur.zip`** or the badge degrades to the open-on-GitHub behavior. Tag format **must use the `v` prefix** to match the existing release history (`v2026.05.20.4`, `v2026.05.20.3`, …) — GitHub's Releases UI groups tags by format and a bare numeric tag will sort oddly on the page. The `v` is stripped before comparison, so the dotted-integer numeric body must match `build-app.sh`'s `VERSION` (e.g. `VERSION="2026.05.20.4"` ↔ tag `v2026.05.20.4`). **To publish an update:** (1) bump `VERSION` in `build-app.sh`, (2) `./build-app.sh` to build + zip, (3) commit, (4) `git tag v<VERSION> && git push origin main v<VERSION>`, (5) `gh release create v<VERSION> --latest --title "..." --notes "..." dist/Murmur.zip dist/Murmur.dmg` (zip feeds the self-updater, dmg feeds murmur.ketok.id's `releases/latest/download/Murmur.dmg` button — ship both). Users running an older build see the badge within 6 hours of next launch. The `swift run` dev binary reports its version as `dev`; `hasUpdate` always returns false there since you can't meaningfully compare a string against a tag.

## Things to leave alone unless explicitly asked

- The `[.titled, .closable, .resizable, .fullSizeContentView]` styleMask combo plus hidden window buttons is what produces the "chromeless" look while keeping resize/drag working. Simpler combos (`.borderless` alone, or omitting `fullSizeContentView`) lose either resize or drag.
- `webView.setValue(false, forKey: "drawsBackground")` is required so the page background (`#0d0d12`) shows through instead of WKWebView's default white flash.
- The `youtube-nocookie.com` origin is intentional (no third-party cookies, fewer overlays). Don't switch to `youtube.com`.
